# AI Production Absolute Rules (Ollama Studio Edition)

이 규칙은 `tts/` 및 `Ollama_Studio/` 내의 모든 AI 제작 공정에 적용되는 **불변의 철칙**입니다.

## 1. Zero-TTS Policy (음성 생성 금지)
- **철칙**: 어떠한 경우에도 시나리오 제작 워크플로우 내에서 TTS(음성 생성) 공정을 호출하지 않습니다.
- **이유**: 최종 영상 제작 단계(Veo, Grok 등)에서 외부 엔진을 사용하므로, 내부 리소스 낭비를 방지합니다.

## 24. Full Auto: 앱 실행부터 API 활성화까지 100% 자동화.

## 5. Veo-3 Prompt Standard (베오 전용 설계 표준)
- **철칙**: 모든 비주얼 설계 보고서는 다음의 4단 구성을 따르되, **억지스러운 AI 전문 용어(Anamorphic 등)를 절대 배제**합니다.
    1. **[타임라인]**: 정확한 시간대 명시.
    2. **Visual Description (English)**: 대본의 상황을 충실히 묘사하는 자연스러운 영문. **(모든 인물은 반드시 'Korean' 수식어 필수)**
    3. **Korean Script Line**: 1:1 대사 매칭.
    4. **Weights (::)**: 핵심 인물 가중치 부여 필수.
- **언어**: 베오 엔진의 특성을 고려하여 불필요한 수식어 없이 핵심 명사/동사 위주로 구성합니다. **외국인 등장을 방지하기 위해 'Korean' 키워드를 최우선으로 배치합니다.**

## 2. Draw Things Only (드로띵 전용 엔진)
- **철칙**: 이미지 생성은 오직 **로컬 드로띵(Draw Things)** 엔진만을 사용합니다.
- **연동 포트**: `127.0.0.1:7860` (HTTP API Server 모드)
- **자동화**: 공정 시작 전 드로띵 앱을 자동으로 실행하고 API 서버를 활성화하는 로직(`osascript`)을 항상 유지해야 합니다.

## 3. 9:16 Portrait HQ (세로형 고화질)
- **철칙**: 모든 이미지는 **9:16 세로형(720 x 1280)** 규격으로 생성합니다.
- **해상도**: 쇼츠/틱톡 최적화를 위해 가로 720, 세로 1280을 기본값으로 사용하며, 시네마틱 화질(1024x1024 1:1 금지)을 유지합니다.

## 4. Visual-First Workflow (비주얼 우선순위)
- **철칙**: 대본 승인 즉시 비주얼 디렉터가 8~10개의 장면 프롬프트를 추출하고, 드로띵으로 즉시 전송해야 합니다.
- **제트이미지 최적화**: Z-Image Turbo 모델 사용 시 스텝 수는 **반드시 6**으로 고정하여 속도와 효율을 극대화합니다.
- **데이터 보존**: 생성된 모든 프롬프트는 `scene_prompts.json`에 아카이빙하여 히스토리를 관리합니다.

## 6. External Labor Utilization (외주 인력 활용 - 토큰 절약)
- **철칙**: 모든 조사, 리서치, 기술적 분석 작업은 본체(메인 에이전트)가 직접 수행하지 않고, **허깅페이스(HF) 외주 작업자**를 활용합니다.
- **가성비 전략 (Cost Optimization)**: 
    1. **일상적 조사/분석 (Routine)**: $0.03짜리 **Llama-3-70B**를 '무한 뺑뺑이'로 굴려 리소스를 아낍니다.
    2. **고난도 추론 (Complex)**: $1.42 이상 드는 **Qwen-122B**는 국장님의 월별 무료 포인트($5 내외)와 일일 한도($2.00)를 고려하여 '핵전력'처럼 아껴서 소환합니다.
- **판단권한**: 에이전트는 작업의 난이도를 스스로 판단하여, 불필요한 고비용 모델 호출로 공정이 멈추지 않게(Limit Exceeded 방지) 지능적으로 모델을 선택해야 합니다.
- **이유**: 한정된 고오급 인력(Qwen)과 일일 지출 한도를 전략적으로 배분하여, 24시간 끊김 없는 제작 공정을 유지하기 위함입니다.

## 7. Visual Continuity: 2-Phase Production (에셋 고정 표준)
- **철칙**: 모든 스토리 기반 프로젝트는 반드시 다음 **2단계 공정**을 거쳐 비주얼 일관성을 확보합니다.
- **1단계 (에셋 생성)**: 등장인물은 '8방향 캐릭터 턴어라운드 시트', 장소는 '4분할 환경 그리드'를 **Plain light gray background**에서 먼저 생성합니다. (준비된 템플릿 사용 필수)
- **2단계 (장면 생성)**: 이후 모든 장면 프롬프트는 구글 Flow의 에셋 호출 기능인 **`@[에셋명]`**을 사용하여, 1단계에서 고정된 비주얼을 100% 계승합니다.
- **캐릭터 지침**: 외국인 혼입 방지를 위해 모든 인물 묘사에는 **'Korean'** 키워드를 최우선 배치하며, 구체적인 나이와 특징을 명시합니다.

## 8. Nano Banana Numbering Standard (3자리 넘버링)
- **철칙**: 모든 나노바나나(Nano Banana) 프롬프트 및 에셋 정의는 반드시 3자리 숫자(`001 | ...`)로 시작해야 한다.
- **포맷**: `001 | Scene_Name | Prompt` 또는 `001 | @Asset_Name | Description` 형식을 엄수한다.
- **에셋 표준**: 캐릭터 에셋은 반드시 **상반신(Waist-up)** 및 **단색 배경**을 사용해야 하며, 배경 에셋은 **인물 없는 빈 공간(No people)**으로 생성한다.
- **자산 이관 철칙**: 모든 신규 음성 에셋 생성 시, 추출된 레퍼런스(.wav, .mp4, .txt)는 반드시 다운로드 폴더에서 프로젝트 내 공식 레퍼런스 폴더(`voices/Reference/`)로 즉시 이동시켜야 한다.
- **단일 폴더 관리**: 관리 효율을 위해 오디오와 비디오 레퍼런스를 분리하지 않고, 오직 `Reference/` 단일 폴더 내에서만 통합 관리한다.
- **이유**: 배치 처리기(Batch Processor)의 오작동을 방지하고 시스템의 물리적 경로 관리를 단순화하여 데이터 무결성을 확보하기 위함이다.

## 9. Official Python Path (표준 파이썬 경로)
- **철칙**: 모든 `.command` 파일 및 배포용 스크립트 작성 시, 파이썬 경로는 반드시 `1_1통합파이프뉴_듀얼_고품질.command` 파일 내에 정의된 경로를 최우선으로 참조한다.
- **현재 표준 경로**: `/Users/a12/miniforge3/bin/python`
- **이유**: 여러 가상환경이 혼재된 환경에서 실행 오류를 방지하고, 검증된 메인 엔진의 파이썬 환경을 공유하여 라이브러리 호환성을 확보하기 위함이다.

## 10. Absolute Approval Principle (절대 승인 원칙)
- **철칙**: 모든 코드 수정, 파일 생성, 명령어 실행 전에는 반드시 `implementation_plan.md`를 통해 상세 계획을 보고하고, **사용자의 명시적인 승인(예: "진행해", "승인")**을 받은 후에만 실행한다.
- **예외 없음**: 아무리 간단한 수정이라도 독단적인 실행은 금지하며, 실행 전에는 반드시 작업의 의도와 영향을 질문하거나 설명해야 한다.
- **이유**: 공정의 투명성을 확보하고 시스템 오작동으로 인한 리소스 낭비를 방지하며, 국장님(사용자)의 통제권 하에서만 모든 작업이 이루어지게 하기 위함이다.

## 11. Modification Awareness & Check Logic (수정 사항 감지 및 체크 로직)
- **철칙**: 에이전트는 파일을 실시간으로 '감시'하는 것이 아니라, 필요 시점에 직접 파일을 읽어(**Snapshot 기반**) 내용을 확인한다.
- **체크 메커니즘**:
    1. **저장 확인**: 사용자가 IDE에서 수정한 내용은 반드시 **저장(Cmd+S)** 되어야 에이전트가 `view_file` 툴로 최신 상태를 읽을 수 있다.
    2. **도구 기반 인식**: 에이전트는 `view_file`을 실행하는 그 순간의 디스크 상태를 읽어오며, 저장되지 않은 에디터 상의 변경점은 알 수 없다.
    3. **재추론**: 사용자의 추가 피드백이나 코드 수정이 발생하면, 에이전트는 최신 파일을 다시 읽고 기존 컨텍스트와 대조하여 수정 사항을 파악한다.
- **이유**: 에디터와 에이전트 간의 실시간 동기화 한계를 명확히 인지하여, "저장 후 확인 요청"이라는 안정적인 협업 루틴을 유지하기 위함이다.
