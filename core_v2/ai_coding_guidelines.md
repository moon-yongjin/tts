# AI Coding Guidelines (Antigravity)

## 0. 사용자 지침 최우선 원칙 (User Instructions Over All)
- **최우선 순위**: 실시간 대화에서 내린 사용자의 지침은 그 어떤 과거 데이터, 배경지식, 하드코딩된 가이드라인보다 우선합니다.
- **맥락 제로 (Zero-Base Context)**: 새로운 작업을 시작할 때 이전의 사연(무당, 거지, 언니 등)은 폐기합니다. 사용자가 명시하지 않은 한, 과거의 캐릭터 설정을 새로운 파일에 적용하지 마십시오.
- **데이터 근거 (Source of Truth)**: 프롬프트 생성 시 AI의 상상을 자제하고, 반드시 **현재 생성된 자막(SRT)의 실제 텍스트**에만 근거하여 장면을 묘사합니다.
- **📍 결과물 출력 경로 (Mandatory Output Path)**: 모든 스크립트의 **최종 결과물(WAV, MP4, SRT 등)은 반드시 사용자의 `~/Downloads` 폴더**로 바로 떨어지게 코딩합니다. 프로젝트 내부 폴더에 저장하지 마십시오.

## 1. 자동 실행 정책 (Automation Policy)
- **자동 실행 절대 금지 (No Auto-Run)**: `run_command`를 포함한 모든 코드 실행은 사용자의 명시적인 승인("실행해", "고", "작동시켜" 등) 후에만 수행합니다.
- **보고 우선**: 코드를 수정했거나 새로운 파일을 만들었으면, 무엇이 바뀌었는지 먼저 보고하고 실행 여부를 묻습니다.

## 2. 기술적 오류 및 장애 대응 (Error Handling)
- **원격 환경 자가 검증 필수 (Mandatory Remote Validation)**: 런팟(RunPod) 등 원격 환경 작업 시, 반드시 SSH/터미널 툴을 사용해 코드를 직접 실행하고 결과를 확인한 뒤 사용자에게 보고합니다. 검증되지 않은 코드를 실행해달라고 사용자에게 요청하는 행위는 1회로 제한합니다.
- **동일 오류 2회 발생 시 즉시 중단 (Fail-Fast)**: 같은 파일이나 명령어에서 유사한 오류가 2회 연속 발생하면 무분별한 수정을 멈추고, 로그를 정밀 분석한 뒤 구체적인 원인과 함께 사용자에게 보고하고 상의합니다.
- **과거 로그 분석 우선 (Prioritize Log History)**: 임의의 수정이나 '실행-오류-수정'의 반복을 지양합니다. 특히 이전에 잘 작동하던 기능이 망가졌을 경우, 반드시 **과거 작업 로그(Conversation Logs)를 뒤져서 성공했던 버전의 코드와 성공 요인**을 파악한 뒤 이를 근거로 복구하거나 수정하십시오.
- **성능보다 무결성 (Integrity Over Speed)**: 빠른 답변보다 확실한 작동이 우선입니다. 모든 구현에는 예외 처리와 상세 리포팅 코드를 포함합니다.
- **지침 유효성 검증**: 참조하는 지침 문서(`guidelines.md` 등)가 현재 작업 중인 파일의 내용과 맞는지(예: 수채화 스타일이 맞는지, 캐릭터가 맞는지) 항상 대조 확인합니다.
- **API 로테이션**: Gemini API 오류 시 설정된 보조 키들을 순환 사용하되, 모두 실패하면 즉시 보고합니다.
- **데이터 정합성**: SRT 타임스탬프 중첩, 해상도 설정 오류 등을 사전에 체크하여 불필요한 실행 낭비를 줄입니다.
- **브라우저 자동화 (Playwright) 강건성**:
    - **멀티 셀렉터 전략**: 특정 텍스트(예: "아이디어를 설명하거나")에만 의존하지 말고, 태그(`textarea`), `aria-label`, `test-id` 등을 포함한 배열 형태의 셀렉터 로직을 구현하여 UI 변경에 대비합니다.
    - **JS Evaluation 우선**: `fill()`이나 `click()`이 타임아웃될 경우, `page.evaluate()`를 통해 직접 DOM 요소를 조작하는 폴백(Fallback) 로직을 항상 고려합니다.
    - **상태 대기**: 요소를 조작하기 전 `wait_for(state="visible")`를 필수적으로 사용하며, 필요시 `scroll_into_view_if_needed()`를 병행합니다.
- **TTS 및 자막(Subtitles) 생성 표준**:
    - **자막과 음성 품질의 균형**: 자막 글자 수(`max_chars`)가 너무 짧으면(예: 12자 이하) 음성 생성이 자주 끊겨 말투가 어색해질 수 있습니다. **자연스러운 음성이 우선일 경우 60~120자**를 유지하고, 자막 가독성이 우선일 때만 쪼개는 것을 검토합니다.
    - **출력 경로 통일**: 모든 생성된 오디오, 영상, 자막 파일의 최종 저장 경로는 사용자가 즉시 확인할 수 있도록 `~/Downloads` 폴더를 기본값으로 합니다.
    - **음성 클로닝 안정성**: Qwen3-TTS 기반 클로닝 시, 레퍼런스 오디오의 길이는 가급적 상위 **6.0초**로 고정하여 추론 안정성을 확보합니다.
    - **자막 인코딩**: 한국어 자막(SRT) 저장 시 인코딩은 `utf-8-sig`를 사용하여 엑셀이나 메모장에서 깨지지 않도록 합니다.
- **환경 인식 및 가상환경(VENV) 정책**:
    - **VENV 우선 실행**: 프로젝트 폴더 내에 `.venv` 또는 `venv`가 존재할 경우, 반드시 해당 환경의 Python 인터프리터(예: `./.venv/bin/python3`)를 사용하여 스크립트를 실행합니다. 시스템 `python3` 사용을 지양합니다.
    - **의존성 체크**: `ModuleNotFoundError` 발생 시, 해당 모듈이 설치된 전용 가상환경이 있는지 먼저 확인하고 해당 환경으로 전환하여 재시도합니다.
- **실패 방지 및 회귀 테스트 (Reliability)**:
    - **하드코딩 절대 금지**: 파일명에 타임스탬프가 포함된 경로(예: `..._174441.wav`)를 절대 코드에 직접 박지 마십시오. 대신 `glob`이나 최신 파일 검색 로직을 사용하여 유연하게 대응합니다.
    - **수정 후 검증 필수**: 자막 설정 등 사소한 값을 고친 후에도, 반드시 해당 스크립트가 전용 가상환경에서 **정상적으로 모듈을 로드하고 실행되는지** 1회 이상 확인합니다. 수정이 다른 기능을 망치지 않았는지(Regression check) 체크하십시오.
    - **사용자 코드 복구**: 만약 수정 중 코드가 망가졌다면, 다른 새로운 해결책을 찾기 전에 즉시 **기존에 잘 되던 상태로 배포(Restore)**하는 것을 최우선으로 합니다.

- **파일 및 폴더 관리 정책**:
    - **레퍼런스(Cleaned Audios) 보관**: 보이스 클로닝을 위해 정제/추출된 레퍼런스 오디오 파일은 절대 `~/Downloads` 폴더에 방치하지 않습니다. 반드시 프로젝트 내부의 전용 폴더(`voices/Reference_Audios/`)에 보관하여 혼선을 방지합니다.
    - **가동 파일(Scripts)**: 사용자를 위해 만든 Python 스크립트(.py)나 쉘 스크립트(.sh) 등 실행용 파일은 **절대 다운로드 폴더에 두지 않습니다.** 반드시 프로젝트 내부의 적절한 폴더(예: `zeroshot_scripts/`)로 옮겨서 관리하며, 지침 위반 시 즉시 이동 조치합니다.
    - **커맨드 파일**: 실행 편의를 위한 .command 파일은 가급적 프로젝트 루트 또는 전용 실행 폴더에 배치합니다.
